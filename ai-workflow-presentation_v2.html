Part 1

The Problem

🤯

What Takes Time in Feature Development?

  • Context switching between layers
  • Remembering patterns and conventions
  • Writing boilerplate code
  • Inconsistency across team members
  • Onboarding new developers
Think about your last feature. How much time did you spend just remembering the right patterns vs actual problem solving?

The Cognitive Load Problem

Files needed:

  • CreditScoreClient.java
  • CreditScoreClientConfig.java
  • CreditScoreClientProperties.java
  • CreditScoreResponse.java
  • CreditScoreClientException.java
  • CreditScoreHealthIndicator.java
  • CustomerFacade.java (update)
  • CustomerController.java (update)
  • CreditScoreClientTest.groovy
  • CustomerFacadeTest.groovy (update)
  • application.yml (update)
  • ...

10-15 files, each with specific patterns to remember

This is a real example. A "simple" feature requires touching many files, each with its own conventions and patterns.

The Solution

💡

AI as a Knowledgeable Pair Programmer

NOT

  • Replacing developers
  • Making decisions for you
  • Writing untested code
  • Magic black box

YES

  • Augmenting developers
  • Encoding team patterns
  • Consistent output
  • Transparent templates
Developers focus on what to build,
AI helps with how to build it

Two Approaches to AI-Assisted Development

Vibe Coding

"Just prompt and pray"

  • One-shot prompts
  • Unstructured requests
  • No planning phase
  • Random patterns

Structured Agents

"Think first, implement in phases"

  • Spec → Story → Plan → Implement
  • Specialized agents per layer
  • Team patterns encoded
  • Consistent output

Vibe Coding: The Reality


Prompt: "Add payment processing feature"

Result: 47 files changed, 3,200 insertions(+)
  • Massive, unreviewable PRs
  • Mixed patterns and conventions
  • No traceability to requirements
  • Teammates can't understand the changes
  • Different output every time
This is what happens when you ask AI to build a feature without structure. The output might work, but can your team maintain it?

Phase 1: Specification

Create a high-level PRD-style specification


/spec PAYMENT "Payment Gateway Integration"

What goes into a spec?

  • Product Overview: Business context and goals
  • Features: What capabilities the system will have
  • Data Models: Entities and relationships
  • API Contracts: Endpoints, requests, responses
  • Technical Decisions: Architecture choices
The spec is like a PRD - it describes the complete feature from a business and technical perspective before any code is written.

Specification Example


# SPEC-PAYMENT-001: Payment Gateway Integration

## Product Overview
Enable real-time payment status checks from external gateway...

## Features
- Query payment status by transaction ID
- Handle multiple payment methods...

## Data Models
### PaymentStatus Entity
- id: PaymentStatusId (composite: transactionId, timestamp)
- status: enum (PENDING, COMPLETED, FAILED)
- paymentMethod: String
- amount: BigDecimal...

## API Contracts
### GET /api/v1/payments/{transactionId}/status
Response:
{
  "transactionId": "TXN123",
  "status": "COMPLETED",
  "completedAt": "2024-01-23T10:30:00Z"
}

Complete picture BEFORE breaking into stories

Why Specification First?

Benefit Explanation
Shared Understanding Everyone knows what we're building
Traceability Link code back to requirements
Context for AI AI understands full picture when generating code
Design Review Catch issues before implementation

The workflow adapts to YOUR process

Phase 2: Story Decomposition

Break specification into implementable user stories


/story PAYMENT-001 "Check payment status via REST API"

What's in a story?

  • User Story: As a... I want... So that...
  • BDD Acceptance Criteria: Given/When/Then scenarios
  • Business Rules: Edge cases and constraints
  • Link to Spec: Traceability to parent document
Stories are the unit of work. They're small, testable, and independently deliverable.

BDD Acceptance Criteria


## Acceptance Criteria

### Scenario 1: Successfully retrieve payment status
**Given** a valid transaction ID exists in the system
**When** client calls GET /api/v1/payments/{transactionId}/status
**Then** return 200 OK with payment details
**And** include status (PENDING, COMPLETED, FAILED)

### Scenario 2: Handle not found
**Given** transaction ID does not exist
**When** client calls the API
**Then** return 404 Not Found

### Scenario 3: Gateway timeout
**Given** payment gateway is slow to respond
**When** request takes longer than 5 seconds
**Then** return 504 Gateway Timeout
**And** log error for monitoring

These drive test cases and agent instructions

Why BDD Format?

  • Testable: Maps directly to test scenarios
  • Unambiguous: Clear inputs and expected outputs
  • Business-Readable: Non-technical stakeholders can review
  • AI-Friendly: Agents understand exact requirements
Good acceptance criteria = Good tests = Good code

Phase 3: Implementation Planning

Generate phased plan with agent assignments


/plan PAYMENT-001

What's in the plan?

  • Phase Identification: Which phases to execute/skip
  • Task Breakdown: Specific files and components to create
  • Agent Assignments: Which agent handles each phase
  • Test Scenarios: Derived from BDD criteria
  • Progress Tracking: Checkboxes for completion

Example Implementation Plan

## Phase Analysis
- ✅ Phase 1: Persistence (new PaymentStatus entity)
- ✅ Phase 2: Integration (PaymentGatewayClient)

## Phase 2: Integration Clients (@integration-client-developer)

Smart Phase Detection

Not every story needs all phases

Story Type Phases
Full stack feature 1, 2, 3, 4, 5 (all phases)
API-only change Skip 1, 2 (no new data/clients)
External integration Skip 1, 4 (no entity/API)
Backend job Skip 4 (no REST API)
Pure CRUD Skip 3 (no complex logic)

Planning phase identifies what's truly needed

  • Lives in .claude/agents/ directory

Anatomy of an Agent


# Persistence Developer Agent

## Purpose
Creates JPA entities, repositories, Liquibase changelogs...

## Scope
**IN SCOPE:**
- JPA entities with proper annotations
- Spring Data repository interfaces
- Liquibase XML changelogs
- Repository tests

**OUT OF SCOPE:**
- REST controllers (use REST API agent)
- Business logic (use Domain Logic agent)

## Code Templates

### Entity Template
```java
@Entity
@Table(name = "{table_name}")
@Data
@EntityListeners(AuditingEntityListener.class)
public class {Entity} implements Persistable<{Entity}Id> {
    @EmbeddedId
    private {Entity}Id id;
    // ...
}
```

This is YOUR team's knowledge, documented and reusable

Why Specialized Agents?

Benefit Explanation
Focus Each agent knows one thing deeply
Quality Templates enforce best practices
Consistency Same patterns across all features
Maintainability Update agent = update all future code

When NOT to Use Agents

  • 🔧 Simple one-line fixes
  • 🔍 Highly exploratory work
  • 📚 When you need to learn the pattern
  • 🎨 When templates don't fit your case
  • ⚡ Quick prototypes

The Review Mindset

Treat AI output like code from a junior developer:

  • ✅ Follows patterns correctly
  • ⚠️ Might miss edge cases
  • ⚠️ Might over-engineer
  • ❓ Business logic needs validation

Always review. Always test. Always think.

These Are YOUR Agents

How to Evolve Agents

  1. Notice repeated pattern in code reviews
  2. Update agent template
  3. All future code follows new pattern

Example:

"We decided to use Optional instead of null returns"


// Update in integration-client-developer.md:
public Optional getById(String id) {
    // ... returns Optional.empty() on 404
}

→ All future clients follow this pattern
